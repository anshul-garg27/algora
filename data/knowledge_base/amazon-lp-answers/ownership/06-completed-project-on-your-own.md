# Q: Tell me about a time where you completed a project on your own.

> **LP**: Ownership
> **Primary story**: `W8 — DC Inventory Search API`
> **Backup story**: `G7 — Sole Architect for 6 Services`
> **Time budget**: 90–120 seconds spoken

---

## STAR — how to actually tell it

### Situation

Walmart suppliers needed a way to query inventory across distribution centres in bulk. Pepsi might want on-hand and in-transit counts for 50 SKUs at DC 6012, all in one call. There was no endpoint for it. Product gave the work to me, and there wasn't another backend engineer available — the consumer team was the consumer team, not co-builders.

### Task

Take it from "we need this" to live in production across US, Canada, and Mexico. Eight PRs, five months, solo on the backend side.

### Action

I forced myself to do the boring thing first — write the spec. Wrote 898 lines of OpenAPI in PR #260 before any code. Endpoints, request and response schemas, validation rules, examples for every error code. The reason wasn't process — it was timing. The consumer team could mock against the spec and start their integration on day one instead of waiting three weeks for my implementation. That single decision cut about 30% off the end-to-end calendar time.

PR #271 was the implementation — about 3,000 lines across 30 files. The shape: a 3-stage pipeline. Stage 1 converts WmItemNumbers to GTINs through UberKey. Stage 2 validates each GTIN against supplier authorisation. Stage 3 calls Enterprise Inventory for actual DC counts. Each stage tracks its own errors. A 100-item bulk request might come back with 80 successes and 20 errors from three different stages — the consumer sees exactly what failed and why.

For multi-country I used a factory pattern. `USConfig`, `CAConfig`, `MXConfig` all `@Component("US")` etc., Spring auto-discovers them as a `Map<String, SiteConfigProvider>`. Adding Mexico was literally one class.

Honestly, the part I'm most proud of is something small. Internally the supplier validation works with GTINs because that's what the supplier mapping uses. But the consumer sent WmItemNumbers. So I reverse-mapped the errors back to WmItemNumbers before returning. Their error message shows their input, not our internal GTIN. Tiny detail, real impact when a supplier engineer at Pepsi is debugging at 11 PM.

Then I caught two more things. Around month three, I realised the validation logic was duplicated between this API and a Search Items API. PR #322 — about 1,900 lines — centralised it into a reusable `RequestProcessor` pipeline. And the GTIN reverse-mapping had a non-determinism bug — some GTINs mapped to multiple WmItemNumbers and my map picked one randomly. PR #330 fixed that with null guards, duplicate logging, and deterministic selection. Then PR #338 — 1,724 lines of container tests with Docker, WireMock, real PostgreSQL — to lock down that whole class of bug.

### Result

Live in US, Canada, and Mexico. Bulk queries up to 100 items per request. About 8,000 lines from spec to container tests across eight PRs over five months. Design-first cut integration time by roughly 30%. Adding the next country is one class.

If I redid it, I'd write container tests in PR #271, not PR #338. The GTIN bug wouldn't have shipped to prod.

## Technical depth — if they probe

- **Design-first over code-first**: Spec is the source of truth. Controller `implements InventorySearchDistributionCenterStatusApi`, an interface generated by `openapi-generator-maven-plugin`. Spec change → compile error → forces controller update. Enforced at compile time.
- **Always HTTP 200 with per-item status**: 207 Multi-Status is technically correct per RFC 4918 but comes from the WebDAV spec and most consumers don't handle it cleanly. 200 with `dataRetrievalStatus = SUCCESS | ERROR` is what every HTTP client already handles.
- **Constructor injection over `@Autowired` fields**: Final fields, immutable dependencies, fail-at-startup if missing. Spring recommends this since 4.3. Tests just pass mocks to the constructor — no reflection.
- **`try-with-resources` for Dynatrace transaction marking**: Each pipeline stage wrapped in `try (var txn = ...addChildTransaction(...))`. If a stage throws, the transaction still closes — no orphan spans in Dynatrace.
- **`@JsonIgnoreProperties(ignoreUnknown = true)` on EI response records**: Enterprise Inventory is upstream of us, we don't control it. If they add a field, we don't break.

## Likely follow-ups

**Q: How did you handle the consumer team's questions while you were still building?**
> The spec answered most of them. I held a 30-minute office hour every week — anyone who'd reviewed the spec could come ask follow-ups. By the time my implementation was done, they were mostly integration-testing already.

**Q: What was the toughest design decision?**
> Whether to return HTTP 207 or always 200 for partial success. 207 is technically correct but I bet on the simpler consumer experience. Each item has its own `dataRetrievalStatus` field. We documented the partial-success pattern in the spec. No consumer has complained.

**Q: Did you ever feel stuck?**
> Yes, in week three. The supplier authorisation framework — Consumer → DUNS → GTIN → Store — was older code with sparse docs. I spent three days reading repos before I understood the four-level check. After that, integration was straightforward.

**Q: How is "completing on your own" different from working solo because no one helped?**
> Big difference. I had reviewers — senior engineers reviewed every PR. I had a tech lead I could ping. I just owned every design call and every defect. Solo on the build, not solo on the company.

**Q: Did you write tests before code?**
> Not strictly TDD. I wrote the OpenAPI spec first, then container tests for the contract (PR #338), then unit tests with each PR. Container tests came late, which is the one thing I'd redo.

## What NOT to say

- Don't claim "no one helped" — code reviewers, tech leads, the consumer team all helped. I owned it; I didn't do it in a vacuum.
- Don't say I wrote tests first — container tests came late, in PR #338.
- Don't pitch this as ML or distributed systems — it's a well-designed REST API on a 3-stage pipeline. Be honest about scope.
- Don't claim 100% test coverage at launch — the GTIN reverse-mapping bug shipped to prod.
