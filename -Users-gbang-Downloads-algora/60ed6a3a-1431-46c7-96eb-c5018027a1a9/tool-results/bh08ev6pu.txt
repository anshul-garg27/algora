310: 
311: Mostly reuses 6.1 — the new behavior is the **atomic claim** and the **shared namespace** with auto-generated codes. The risk is two creators racing for the same alias, or an alias colliding with a future auto code.
312: 
313: **Architecture decisions table**
314: 
315: | Component | What it is (plain English) | Why THIS choice | What we DIDN'T pick & why not | Trade-off we accept |
316: |---|---|---|---|---|
317: | Alias claim (unique PK insert) | The mapping insert uses the alias as the primary key; DB rejects a duplicate | DB-enforced uniqueness is the simplest correct serialization point | Not "read-then-write in app code" (classic race: both read free, both write) | A loser just retries with a different alias |
318: | Namespace partition | Auto codes are base62 of large counter IDs; reserved-word denylist for aliases | Keeps custom and auto codes from colliding by construction | Not a shared random pool (then auto and custom can clash) | Aliases can't look like a future auto code prefix — minor UX rule |
319: 
320: > 🎙️ **Narration:** "Custom aliases share the same code namespace as auto codes, so I need the claim to be atomic. The classic mistake is read-then-write: two users both check 'is it free?', both see yes, both write, one silently clobbers the other. To stop a custom alias ever colliding with a future auto code, auto codes always come from large counter integers and aliases live with a reserved denylist, so the two pools don't overlap."

322: > **🗣️ Key terms for this slice:**
323: > **Unique constraint** — a DB rule that no two rows share the same key. Here the alias is the PK, so a duplicate insert simply fails instead of overwriting.
324: > **Namespace collision** — when two different generation strategies produce the same code. Here we prevent it by keeping auto codes in the high-integer space and aliases gated by a denylist.

437: 
438: **PART B — Storage decision cards**
439: 
440: **🗄️ Postgres (sharded) — used for: ShortLink, Owner, Counter**
441: 1. **What it is:** a relational database that stores rows with strict rules and guarantees a write either fully succeeds or fully rolls back; here it enforces that a code is unique.
442: 2. **Why we picked it:** we need an **atomic insert-if-absent** so two creators racing for the same custom alias can't both win, and a strong PK uniqueness guarantee — only a relational DB gives this cleanly. Writes are light (~6K/s), so a sharded Postgres is plenty.
443: 3. **What we considered instead:**
444:    - **DynamoDB** — great KV for the read path, but its conditional writes work; the real reason against it as the *source of truth* is we'd still bolt on a counter scheme and we prefer Postgres's transactional ergonomics + cheaper at this write rate.
445:    - **Cassandra** — excellent write throughput, but last-writer-wins by default makes the alias-uniqueness guarantee awkward (lightweight transactions are slow).
446:    - **One unsharded MySQL** — identical capability to Postgres but 375TB won't fit on one node; we'd shard either way and prefer Postgres tooling.
447: 4. **Trade-off we accept:** sharding by code means cross-shard queries (e.g. "all links by owner") need a secondary index or a separate owner→codes table; we denormalize.
448: 5. **🗣️ How to say it:** "Mappings live in Postgres sharded by code, because custom aliases need an atomic claim and I want database-enforced uniqueness, not application locks. I looked at Dynamo and Cassandra — fine for the read path — but the uniqueness guarantee is cleanest in Postgres. The cost is cross-shard owner queries, which I solve with a small owner→codes table."
449: 
450: **🗄️ Redis (sharded) — used for: hot code → long_url cache**
451: 1. **What it is:** an in-memory key-value store with sub-millisecond reads.
452: 2. **Why we picked it:** the read path is ~580K/s and the DB can't do that; an in-memory cache sharded across ~4 nodes absorbs it at sub-ms latency.
453: 3. **What we considered instead:** **Memcached** — simpler but no persistence/replication if we want warm restarts; **local in-process cache** — can't be shared across many redirect servers and is cold on every deploy.
454: 4. **Trade-off we accept:** cache and DB can briefly disagree — fine because mappings are immutable after create (no update path to go stale).
455: 5. **🗣️ How to say it:** "Redis holds the hot mappings in memory, sharded because one node tops out near 150K ops a second and I'm at half a million. Mappings never change after creation, so cache staleness basically can't happen."
456: 
457: **🗄️ Kafka + OLAP + object store — used for: ClickEvent / analytics**
458: 1. **What it is:** Kafka is a durable replayable log; the OLAP store is a columnar DB built for aggregate scans; the object store (S3/Glacier) is cheap cold archival.
459: 2. **Why we picked it:** at 10B clicks/day we cannot write a row per click to the OLTP DB; a log buffers the firehose, a stream job aggregates, and the columnar store answers "clicks by country" fast.
460: 3. **What we considered instead:** **Postgres per-click insert** — melts at 10B/day; **direct OLAP insert per event** — too many tiny writes, no buffering/replay if the consumer dies.
461: 4. **Trade-off we accept:** eventual counts (seconds of lag) and an extra streaming system to operate.
462: 5. **🗣️ How to say it:** "Clicks ride Kafka into a Flink job that keeps counts in a columnar store, and raw events spill to cheap object storage. I never touch the main DB on a click. The price is a few seconds of lag, which is exactly fine for a dashboard."
463: 
464: **PART C — Per-operation consistency**
465: 
466: | Operation | Store | Consistency | Why right here |
467: |---|---|---|---|
468: | Create link / claim alias | Postgres | Strong (atomic unique insert) | Money-adjacent correctness: a collision sends users to the wrong site |
469: | Counter allocate range | Postgres/etcd | Strong (atomic increment) | Two servers must never get the same ID range |
470: | Redirect read (cache hit) | Redis | Eventual (but effectively strong) | Mappings are immutable post-create, so "stale" can't happen |
471: | Redirect read (miss) | Postgres replica | Read replica (ms lag) | Mapping won't change, so lag is harmless |
472: | Read-your-own-writes (just-created link) | Redis/replica | Handled explicitly | After create, the code may not be in cache/replica yet → see below |
473: | Click count read | OLAP | Eventual (seconds) | Dashboard tolerates lag |
474: 
475: **Read-your-own-writes:** right after creating a link, the creator might immediately click it before the cache or replica has it. Mitigation: on create we **write-through into Redis** and the create response *is* the proof of existence; if a redirect misses both cache and replica within the replica-lag window, the Redirect Service **reads the primary** for that one lookup. Staleness bound: ≤ replica lag (single-digit ms).
476: 
477: **PART D — Data lifecycle (largest table = ClickEvent, ~270TB raw at 90 days)**
478: 
479: Raw click events are the fastest-growing table. We keep **90 days of raw events** (tying to the ~270TB / replication-3 figure from §3); beyond that we keep only **rolled-up aggregates** (counts per code/day) which are tiny. Reclamation is **time-partitioned tables with a daily PARTITION DROP** — dropping a day's partition is an O(1) metadata operation, instant, versus a row-by-row DELETE that churns indexes and never cleanly reclaims space. Cold raw events spill to **S3/Glacier** for replay/audit. Without this we'd grow unbounded and the analytics store would dominate cost.
480: 
481: > 🎙️ **Script:** "The core is Postgres sharded by code — I chose it for database-enforced uniqueness on custom aliases, since a collision is a correctness disaster. The hot reads are absorbed by sharded Redis, and because mappings are immutable I don't worry about cache staleness. Clicks ride Kafka into Flink and land in a columnar analytics store, with raw events kept 90 days then partition-dropped and archived to S3. I considered Cassandra for the source of truth but its last-writer-wins makes the alias guarantee awkward, so I kept the source of truth relational and used the NoSQL-style stores only where eventual consistency is acceptable."

489: ### How do we generate short codes that never collide?
490: 
491: **#### Bad: Random base62 + check-and-retry**
492: - 🗣️ **Plain words:** roll a random 7-character code, ask the DB "is this taken?", and reroll if it is.
493: - **Approach:** generate 7 random base62 chars, `SELECT` to check existence, insert if free, else retry.
494: - **Why people try this:** it's the obvious first idea — no coordinator needed.
495: - ⚠️ **What breaks:** every write now needs an extra read, and as the keyspace fills, collisions (and retries) climb; under a write burst the retry loop amplifies load on the exact DB you're trying to protect. At ~6K writes/s the extra read isn't fatal yet, but it's an unbounded-retry latent risk and it makes uniqueness probabilistic.
496: - 🔁 **Forces upgrade:** the need for *guaranteed* uniqueness with no per-write read pushes us to a counter.

498: **#### Good: Single global counter + base62**
499: - ↩️ **What the previous tier got wrong:** random codes need a DB check on every write; a counter is unique *by construction*, so no check, no retry.
500: - 🗣️ **Plain words:** hand out an ever-increasing number like a deli ticket and base62-encode it.
501: - **Approach:** a single atomic counter (a row with `FOR UPDATE`, or a Redis `INCR`) gives the next integer; base62-encode it into the code.
502: - **Why people try this:** it's correct and simple to reason about.
503: - ⚠️ **What breaks:** the counter is now a **global coordinator** — every create serializes through it. At ~6K writes/s a single `INCR` is fine, but its *availability* is now the write tier's availability, and codes are **sequential/guessable** (you can enumerate other people's links).
504: - 🔁 **Forces upgrade:** the single point of serialization + guessability pushes us to ranged allocation + obfuscation.

516: **#### Great: Ranged KGS + per-server blocks (+ optional bit-permutation)**
517: - ↩️ **What the previous tier got wrong:** a per-write counter call serializes every create and is a single availability dependency; granting *ranges* removes ~99.9% of those calls, and permuting bits removes guessability.
518: - 🗣️ **Plain words:** the counter hands each server a block of a thousand numbers at once, so servers mint codes locally and barely ever talk to the counter; then we scramble the bits so codes aren't sequential.
519: - **Approach:** the **KGS** atomically advances the counter by 1,000 and grants `[N, N+999]` to a server. The server mints codes locally from its block. To kill guessability, we apply a reversible **bit permutation** (a Feistel/XOR scramble — a deterministic two-way shuffle of the integer's bits) *before* base62 encoding, so consecutive IDs produce non-adjacent codes. Custom aliases bypass the counter and use insert-if-absent in the same namespace partition.
